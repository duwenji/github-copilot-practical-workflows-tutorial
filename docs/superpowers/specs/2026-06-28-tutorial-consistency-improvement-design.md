## 背景と目的

`GitHub + GitHub Copilot 実務シナリオ完全チュートリアル` は v0.1.0 時点でスケルトン状態のページが多く、以下の問題がある。

- ページごとにセクション構成がバラバラ（必須情報が欠落しているページが多い）
- 文体のばらつき（ですます／である混在、箇条書き末尾の不統一）
- 内容が極端に薄いページが複数存在する（FAQ 3 行、CI 基礎 20 行など）

**ゴール:** 個人学習者（初〜中級開発者）とチーム導入担当者（Tech Lead / 管理者）の両方が使える、一貫した品質の教材にする。

---

## 設計 1: ページ種別とテンプレート

### 種別 A: 解説ページ（concept / how-to）

概念説明・手順案内を目的とするページ。`docs/01〜06` 以下の大部分が該当する。

**必須セクション（この順序を守る）:**

```
# ページタイトル
## 典型シナリオ          ← どんな場面で使うか（1〜3 文）
## コンセプトと仕組み    ← なぜそうするかの理由
## 基本手順              ← 番号付き手順 + コードブロック
## コマンドと引数の意味  ← コマンドがある場合のみ
## Copilot の使いどころ  ← 具体的なプロンプト例を含む
## 注意点                ← 実務で踏みやすい落とし穴
## 章末チェック          ← 読んだ後に確認できる 2〜3 項目
```

**最低ライン:** 60 行以上、必須セクション 7 つすべて埋める。

### 種別 B: リファレンスページ（cheatsheet / checklist / FAQ）

`docs/07-reference-and-scenario-playbook/` 以下のチートシート・FAQ・チェックリスト。

**必須セクション:**

```
# ページタイトル
## 概要              ← 何のためのリファレンスか（1〜2 文）
## [コンテンツ本体]  ← チートシートなら表、FAQ なら Q&A、チェックリストならチェック項目
## 関連ページ        ← 詳細を知りたい場合のリンク先
```

**最低ライン:** 30 行以上、「関連ページ」リンクを必ず含める（該当リンクがない場合は `index.md` へのリンクを最低 1 つ置く）。

## 設計 2: 文体ガイド

### 文末・文体

| ルール | 詳細 |
|---|---|
| 本文はですます調に統一 | 「〜します」「〜できます」。「〜する」「〜だ」は使わない |
| 箇条書き末尾を統一 | 体言止め または「〜すること」。「〜できる」「〜する」は使わない |

### コードブロック

| ルール | 詳細 |
|---|---|
| シェル種別を統一 | Windows 環境前提のため ` ```powershell ` に統一。` ```bash ` は使わない |
| コマンド説明を必須化 | コードブロックの直後に必ず「コマンドと引数の意味」セクションを置く |

### Copilot プロンプト例

| ルール | 詳細 |
|---|---|
| 文末を統一 | 「〜してください」で統一 |
| 文脈を含める | 抽象的な指示ではなく、プロジェクト・ファイル・場面を含む具体例にする |

**例（修正前）:** 「このコードを説明して」  
**例（修正後）:** 「この `filterQuiz` 関数が何をしているか、3 行以内で説明してください」

### 注意点・チェック項目の粒度

| ルール | 詳細 |
|---|---|
| 行動レベルで書く | 「〜しないこと」または「〜すること」の形に統一 |
| 曖昧な表現を避ける | 「テストを忘れない」→「PR 作成前にローカルでテストを実行すること」 |
