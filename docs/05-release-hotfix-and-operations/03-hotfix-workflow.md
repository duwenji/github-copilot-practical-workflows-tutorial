# hotfix 対応フロー

## 典型シナリオ

本番障害や重大バグに対して、通常より短いサイクルで修正を届ける必要がある場面です。

## コンセプトと仕組み

- `hotfix` は、本番環境の障害を最小限の変更で迅速に解消するための緊急フローです。
- `main` から `hotfix/*` ブランチを切り、最小修正を加えてリリースします。
- 通常の feature ブランチと異なり、review と test を短時間で集中して実施します。
- 適用後は必ず `main` にも反映し、記録を残すことが重要です。
- 速度を優先しつつも、レビューと記録を省略しない運用が求められます。
- インシデント対応中は、Copilot を活用して素早く原因仮説を整理できます。

## 基本手順

1. 影響範囲を確認すること
2. `main` から `hotfix/*` ブランチを切ること
3. 原因を調査して最小修正を入れること
4. テストとレビューを短時間で実施すること
5. release / deploy すること
6. `main` に変更を反映すること
7. 事後確認と記録を残すこと

## 緊急度の判断基準

- 本番ユーザーへの直接影響があるか
- データ破損のリスクがあるか
- 外部サービスや連携先に影響が波及しているか
- 通常の修正サイクルで対応できない緊急性があるか

## Copilot の使いどころ

- 原因仮説の整理
- 最小修正案の比較
- 回帰テスト観点の洗い出し
- インシデント報告の下書き

## hotfix 後の対応

- `main` への反映が完了したことを確認すること
- 事後確認として本番環境の動作を検証すること
- インシデントレポートを作成して関係者と共有すること
- 再発防止策を次のスプリントに組み込むこと
- CI が正常に戻ったことを確認すること

## 注意点

- 緊急でもレビューと記録は省略しないこと
- 追加変更を混ぜ込まないこと
- hotfix 後は通常フローへの影響を確認すること
- インシデントの原因と対策を記録として残すこと

## 章末チェック

- hotfix ブランチの作成から deploy までの流れを説明できること
- 最小修正の範囲を判断できること
- インシデント報告の下書きを Copilot と一緒に作成できること
- hotfix 後の `main` への反映と事後確認を実施できること
