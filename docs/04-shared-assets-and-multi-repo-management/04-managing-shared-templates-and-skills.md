# shared templates / skills を複数 repo で管理する

## 典型シナリオ

複数のリポジトリで、共通のテンプレート、スクリプト、ドキュメント、skills を再利用したい場面です。

## コンセプトと仕組み

- 共有資産管理の本質は、**一元更新しやすさ** と **利用側の扱いやすさ** のバランスです。
- 更新頻度が高いほど中央管理の価値が上がり、利用者が多いほど運用の単純さが重要になります。
- つまり、技術選定は「どの方法が強いか」よりも「どの方法がチームに定着しやすいか」で考えるのが実務的です。

## 代表的な選択肢

- `submodule` で外部参照する
- `subtree` で取り込む
- テンプレート repo からコピーして運用する
- package 化して配布する

## 判断の観点

- 更新頻度
- 利用側の分かりやすさ
- バージョン固定の必要性
- 管理者の運用負荷

## Copilot の使いどころ

- 選定理由の文書化
- 導入手順のたたき台作成
- 更新フローのチェックリスト化
